The Problem We Solve

Every deal has a path to close. ML tells you if that path is breaking.

Homebuilders manage 50-500 active deals simultaneously across 6+ systems. Each deal has ~51 milestones that must complete before closing day. The ML layer answers three questions for every deal, every day:

Will this deal close on time?

The Closing Confidence Score (0-100) predicts the probability that a deal closes by its target date. Drives the traffic-light system in the portal: HEALTHY / AT_RISK / CRITICAL.

Is this deal stuck?

Anomaly detection flags deals with unusual patterns — even when they haven't formally crossed a risk threshold yet. Catches "silently dying" deals before they become emergencies.

WHY is it at risk?

Every prediction comes with ranked explanations: "This deal is at risk because mortgage has been silent for 12 days and construction is 40% ahead of lending." Actionable, not a black box.

ML is now the ONLY scoring engine. The old rule-based calculator (static category weights) is disabled. If the ML service is unavailable, scoring returns "unavailable" — it does NOT silently fall back to a worse model. We fail loud.

The Models

Two deployed models. Each answers a different question.

Model 1: XGBoost Multi-Class Classifier

What it is: A gradient-boosted decision tree. Learns patterns from historical deals and their outcomes. Proven best-in-class for structured/tabular data problems like ours.

What it predicts: Given a deal's current state (features), it outputs three probabilities:
• P(on_time) — probability deal closes by target date
• P(delayed) — probability deal closes, but late
• P(cancelled) — probability deal falls through entirely

Why XGBoost: Compatible with SHAP explainability (we can show users WHY). Handles non-linear relationships between features. Industry standard for tabular ML.

ACTIVE · PRODUCES THE CONFIDENCE SCORE

Model 2: Isolation Forest (Anomaly Detector)

What it is: An unsupervised model that learns what "normal" deals look like. It doesn't need labels — it identifies deals that deviate from the norm.

What it detects: Deals with unusual feature patterns — e.g., a deal that has high construction progress but zero mortgage activity, or one that suddenly stopped receiving signals from all systems.

Why Isolation Forest: No labeled anomalies needed to train. Works alongside XGBoost — sometimes a deal isn't "at risk" by the classifier but IS anomalous (early warning signal).

ACTIVE · FLAGS UNUSUAL DEALS

Baseline tested: Logistic Regression (simple linear model) was trained on the same data to ensure XGBoost adds real value over a trivial approach. On synthetic data, LR actually scored 2% higher (78% vs 76%) — expected with small data. XGBoost is retained because it will outperform at production scale and enables SHAP explanations. This gap must close when we move to real data.

Model Performance

How good is the model at predicting deal outcomes?

Trained on 915 synthetic deal observations. The numbers below mean: if we look at 100 deals, how many does the model classify correctly?

Overall Performance

76% accuracy — out of 183 test deals, the model correctly classified 139.

75.3% cross-validated — trained 5 times on different data splits, consistently performs at ~75%. Low variance (±2.6%) means the model is stable, not getting lucky on one split.

F1 = 0.73 — balances precision (of the deals I flagged as delayed, how many actually were?) and recall (of all delayed deals, how many did I catch?). 0.73 means reasonable on both.

Feature Engineering

What data does the model look at to make predictions?

21 features are computed for each deal, drawn from signals flowing in from all 5 builder systems. Think of these as the "vital signs" of a deal.

Category Completion (6 features)

"How much of each category is done?"

• construction: 5/12 milestones complete = 0.42
• mortgage: 3/10 = 0.30
• title: 0/8 = 0.0
• financial, sales, utility (same logic)

Used by: XGBoost (outcome prediction) + Isolation Forest (anomaly detection)
Source systems: BuildPro, Encompass, Qualia, JDE, Salesforce

Signal Velocity (3 features)

"How active is this deal right now?"

Count of ALL signals received in the last 7 / 14 / 30 days. A deal getting 15 signals/week that drops to 0 is a red flag.

Used by: XGBoost + Isolation Forest
Top predictor: signals_last_30d is the #1 SHAP feature — signal velocity is more predictive than completion percentage.

Days Since Last Activity (6 features)

"How long has each category been silent?"

Calendar days since the last signal per category. A construction gap of 3 days is normal; 21 days means something is wrong.

Used by: XGBoost + Isolation Forest
Source systems: Each category maps to a source system

Time Pressure + Cross-System + Metadata (6 features)

• days_to_target_close: urgency (negative = past due)
• days_since_contract: deal age
• construction_mortgage_ratio: are these two in sync?
• price_tier: deal complexity proxy
• community_avg_velocity: peer comparison
• division_deals_active: workload context

Used by: XGBoost + Isolation Forest

Design Decision

Staleness: a system health signal, NOT a prediction feature.

We extract 5 additional staleness flags (stale_salesforce, stale_buildpro, stale_encompass, stale_qualia, stale_jde). These are deliberately excluded from the ML model.

Why Excluded From the Model

Staleness is a data pipeline problem, not a deal problem.

If Salesforce hasn't sent data in 8 business days, that could mean:
• Our connector is broken
• The builder's Salesforce instance is down
• A weekend + holiday

None of these predict whether the DEAL will close on time. If we trained the model on staleness, it would learn "when data stops flowing, deals fail" — but that's an artifact of our system, not a real signal about the home.

What We DO With Staleness

Alert the user. Staleness flags are exposed via the /features endpoint and consumed by:

• Pattern Detector: "3+ homes with stale Encompass data" → systemic integration issue
• Next Best Action: prepends "Data may be stale from BuildPro" to action recommendations
• Portal: shows freshness warnings when a source is silent

This is the user's responsibility to resolve — they need to ensure data is flowing. We surface the problem; we don't hide it inside a score.

Total features extracted: 26 (21 to model, 5 to alerting). The model only sees real deal health signals. Data pipeline issues are surfaced separately as operational warnings.

Model Lifecycle

How new models replace old ones

When we retrain with new data, here's the process for swapping the active model. This is critical — a bad model in this domain means a VP ignores a failing deal or panics about a healthy one.

Training Pipeline (Offline — runs on demand)

1. Extract: python -m prediction.training_pipeline
   Queries all homes from PostgreSQL
   Extracts features at 25%, 50%, 75% of lifecycle
   Produces Parquet file with labels

2. Train: python -m prediction.models.train
   Trains XGBoost + Isolation Forest + LR baseline
   80/20 stratified split + 5-fold cross-validation
   SHAP feature importance computed
   Outputs: .json model + .joblib artifacts + training_report.json

3. Validate: Compare metrics against previous model
   F1 macro must not regress
   Per-class recall must not drop >5%

Deployment (Model Swap)

Current process:
• New model artifacts placed in /app/models/
• Prediction service restarted (picks up new model at startup)
• model_version field updates automatically from training_report.json
• Portal shows which model version produced each score

Criteria for replacing the current model:
• Must exceed current F1 macro (0.7315)
• Must improve delayed-class recall (currently 60% — the weak point)
• Must pass on the same cross-validation stability threshold (<3% std)
• Must not change anomaly detection rate by more than ±5%

Not yet built: Shadow scoring (running new model alongside old), automated drift detection, and canary rollout. These come when we have enough real data to justify the infrastructure.

Failure Behavior · Critical Review

What happens when ML is unavailable?

This domain is land — deals worth $300K-$1.5M. A VP who trusts a score that silently came from a worse model makes bad decisions. Our policy: fail loud, never silently degrade.

Current Behavior (Correct)

USE_ML_SCORING = true
DISABLE_RULE_BASED_SCORING = true

When ML service is reachable: → ML prediction returned
When ML service is DOWN: → Returns scoring_unavailable = true, score = 0, risk_level = CRITICAL

The portal sees scoring_unavailable and shows a clear "Scoring temporarily unavailable" state — NOT a fake number.

This is the right design. A Division VP seeing "unavailable" will call operations. A VP seeing a stale rule-based number might make a $500K decision on bad data.

What We Removed (and Why)

The old "graceful degradation" pattern:
• ML down → fall back to rule-based weights
• Rule-based is 35/25/20/5/10/5 static allocation

Problem: A rule-based score of 72 and an ML score of 72 mean completely different things. The rule-based number has no predictive validation. Mixing them silently is dangerous.

Decision: Rule-based fallback is DISABLED. If someone re-enables it, they're accepting liability for decisions made on unvalidated numbers.

Open risk: If the ML service has extended downtime (>1 hour), the entire pipeline shows "scoring unavailable" for all deals. We need alerting + fast restart, not a silent fallback.

Score Derivation

How probabilities become a Closing Confidence Score

XGBoost outputs three probabilities that sum to 1.0. We convert these into a single 0-100 score that builders understand.

The Formula

score = P(on_time) × 100
        - P(cancelled) × 30
        - P(delayed) × 10

clamped to [0, 100]

Why asymmetric penalties?
A cancelled deal is a total loss ($300K+ revenue gone). A delayed deal still closes — it's a cost and reputation hit, but recoverable. Cancellation risk gets 3x the penalty.

Risk Level Thresholds

Score ≥ 75 → HEALTHY — on track
50 ≤ Score < 75 → AT_RISK — needs attention
Score < 50 → CRITICAL — intervention required

What Users See Per Prediction

• The confidence score (the number on the dashboard)
• The risk level (the color/badge)
• Top-5 risk factors with plain-English labels: "Mortgage activity has been silent for 12 days" rather than "days_since_last_mortgage = 12"
• Anomaly flag if Isolation Forest fires
• Velocity trend (improving/steady/declining)

Latency

Single prediction: <500ms (p95). Includes DB query for features + XGBoost inference + SHAP computation for top-5 factors. Batch skips SHAP (too expensive for N deals) — factors populated on drill-down only.

Availability Requirement

If this service is down, the portal shows "scoring unavailable" for ALL deals. This makes it a critical-path service. Current mitigation: Docker restart policy + health checks. Future: redundancy.

Gaps & Roadmap

What's missing. What comes next.

Known Gaps (Must Fix)

1. Trained on synthetic data only.
915 samples from a simulator. The model has never seen a real deal. Performance on real data is unknown. Priority #1.

2. Delayed-class recall is 60%.
We miss 40% of deals that will end up delayed. For builders, a false "HEALTHY" on a deal that slips is worse than a false alarm. Must improve with real data.

3. No automated retraining.
Model swap is manual. No drift detection. No shadow scoring. No canary deploys.

4. Anomaly detection not yet integrated with escalations.
Isolation Forest runs and flags, but Task 032 (auto-escalation from anomaly scores) is still backlog.

5. No model monitoring/observability.
No tracking of prediction distribution drift over time. No alerting when model confidence drops.
