Architecture

Three-layer platform — intelligence above existing tools.

Key principle: Never writes back to builder systems. Read-only integration enforced by EgressGuard (blocks non-GET/HEAD requests at the connector level).

This is the three-layer architecture. At the bottom, Layer 1, are the builder's existing systems — BuildPro, JDE, Encompass, Qualia, Salesforce, Procore. We don't replace any of them. We connect to them read-only. Layer 2 is our core product — the AI Transaction Execution layer. Data flows from their systems through our YAML-based connectors into Kafka, then into the Graph Engine where we maintain a dependency DAG per deal. The ML model extracts 26 features and produces a confidence score. All of this serves the API and the builder portal. Layer 3, shown dashed because it's planned, is the blockchain layer — Hyperledger Fabric for immutable event anchoring. It creates a permanent Home Passport that travels with the property. The key principle is that read-only guarantee. We have an EgressGuard that physically blocks any non-GET request at the connector level. Builders can trust we'll never touch their data.

Critical Path Calculation

Kahn's topological sort + longest-path DP identifies which milestones gate the close date. Informed by cross-system dependencies.

This shows the home's lifecycle as we track it. There are parallel tracks running simultaneously: construction with 12 milestones over about 145 days, mortgage with 10 milestones running in parallel, title with 8, sales, financial, and utility. The critical insight is these tracks run in PARALLEL but they BLOCK each other. You can't close without clear-to-close from mortgage AND title clear AND CO issued from construction. If any one track stalls, it blocks the close. We maintain this as a directed acyclic graph per deal, with 3 templates: standard (45 milestones, 48 edges), cash buyer (no mortgage track), and condo (HOA instead of utility). Kahn's topological sort plus longest-path dynamic programming gives us the critical path — which milestones actually gate the close date right now.

This is the ETL pipeline. Step 1: our YAML-driven connectors extract data from builder systems via webhooks, REST polling, or event streams. Step 2: we normalize raw events into our 51 typed signals with deduplication. Step 3: signals flow through Kafka, keyed by deal ID, with a dead-letter queue for failures. Step 4: the Graph Engine routes each signal, updates the DAG, recalculates critical path, and emits downstream events. On top of this sits the analytics layer — the confidence score calculator that weights categories at 35% construction, 40% mortgage, 25% title, minus staleness penalties. And the escalation engine that fires alerts when scores drop or milestones stall. All of this is backed by guarantees: idempotent processing, read-only enforcement, Kafka replay for recovery, and 10 formal correctness properties validated by property-based tests.

Layer 3 is the blockchain — and I want to be clear about our philosophy here: blockchain is infrastructure, not a feature. Builders never see it. They experience trusted workflows and permanent records. Under the hood, Hyperledger Fabric — a permissioned consortium blockchain with no gas fees and role-based access. The two-layer trust model is key: PostgreSQL holds mutable state — deals can go backward, milestones can regress. The blockchain receives append-only event anchors asynchronously. It creates a Digital Home Passport at contract signing and anchors every verified milestone as an immutable timestamped event. This passport travels with the home through ownership changes — containing construction history, mortgage records, warranties, everything. The Land Passport extends this backward to land acquisition in Phase 3. Status: architecture confirmed, dev network deferred. Nothing on our critical path depends on it — we built it to be async so it never adds latency to operations.
